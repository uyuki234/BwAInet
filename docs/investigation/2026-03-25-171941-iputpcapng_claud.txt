  実際に動作していたもの

  1. DHCP — 正常にIP取得完了（Discover → Offer → Request → ACK）
  2. ARP — ゲートウェイ 10.64.56.1 はARP応答を返している
  3. DNS — 10.72.1.11
  は名前解決に正常応答している（外部ドメインも解決成功、rcode=0）
  4. PAC (proxy.pac) — 10.72.1.61/proxy.pac
  を取得し、プロキシ自動設定が適用されている

  通信できない原因：認証プロキシ

  このネットワークは proxy.pac によるプロキシ強制環境 です。

  - プロキシ: proxy01.osaka.hal.ac.jp:8080 (10.76.3.62)
  - proxy.pac のデフォルトルールは return PROXY; — つまり DIRECT_LIST
  にないドメインは全てプロキシ経由が必須
  - プロキシは HTTP 407 (Proxy Authentication Required)
  を返している（Basic認証、33回確認）
  - IPv6接続テストは HTTP 504 (Gateway Timeout) を返している（23回確認）

  ping が通らない理由

  - ゲートウェイ 10.64.56.1 は ICMP Echo Replyを返さない設定
  になっている（ARP応答はあるのでL2接続は確立済み）
  - キャプチャ中、1.1.1.1 への ping が1回だけあるが応答なし — ICMP 
  がフィルタされている

  dig が使えない理由

  - 内部DNSサーバ 10.72.1.11 は実際にはDNS応答を返している
  - ただし dig コマンドはプロキシを経由しないため、ネットワークポリシーによって
  直接の外向きUDP/53が遮断されている可能性がある
  - あるいは、Windows側でDNS resolver経由のクエリは成功するが、dig
  が直接参照するDNS設定と異なる可能性がある

  WireGuard 接続の試行

  - 58.188.75.42:51820 への WireGuard
  接続を56回試みているが、全て一方向（送信のみ）で応答なし — UDP 51820
  が外向きブロックされている

  ネットワーク機器

  - Realtek製スイッチ が多数存在（MAC e0:1a:ea:*、4983フレーム）—
  L2のスイッチ管理プロトコル
  - HP製スイッチ も検出（MAC 10:4f:58:*, 38:10:f0:*）— HPSW プロトコル
  - 典型的な教育機関のマネージドスイッチ構成

  まとめ

  ┌────────────────┬─────────────────────────────────────────────────────────┐
  │      症状      │                          原因                           │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ ping不通       │ GWがICMP無効 + 外向きICMPフィルタ                       │
  ├────────────────┼─────────────────────────────────────────────────────────┤

  │                │ 直接通信はファイアウォールで遮断                        │

  └────────────────┴─────────────────────────────────────────────────────────┘

  対策

  当日ネットワークを利用するには：
  1. プロキシ認証のクレデンシャルをHAL大阪側に確認する（proxy01.osaka.hal.ac.jp:
  8080、Basic認証）
  2. プロキシ設定をブラウザ/OS/アプリに適用するか、proxy.pac
  (http://10.72.1.61/proxy.pac) を自動設定する
  3. ping/dig等のプロキシ非対応ツールはこの環境では使えない前提で準備する
  4. WireGuard VPN はこの環境からは利用不可（UDPが直接通らない）

● はい、キャプチャの証拠からはそう判断できます。

  プロキシが解除されていない明確な証拠

  1. proxy.pac が配信中 — http://10.72.1.61/proxy.pac
  が200で正常に返されており、中身のデフォルトルールは return 
  PROXY;（全通信をプロキシ経由に強制）
  2. プロキシが認証を要求中 — proxy01.osaka.hal.ac.jp:8080 が HTTP 407
  を33回返している
  3. 直接の外向き通信がブロック — ICMP、WireGuard (UDP 51820)
  など、プロキシを経由しない通信は全て応答なし

  ただし一部のDIRECT通信は成功している

  proxy.pac の DIRECT_LIST に含まれるドメイン（Microsoft、Google、Adobe、Unity、
  Steam等）はプロキシを迂回して直接通信が成立しています。キャプチャ中にTLS接続が
  多数成功しているのはこのためです。

  つまり、ネットワーク的にはインターネットへの経路自体は存在するが、PACファイル
  とファイアウォールの組み合わせにより、DIRECT_LISTに載っていないドメインはプロ
  キシ経由を強制され、その認証で弾かれている状態です。

  「解除」が何を意味していたか次第

  考えられるパターン：

  ┌──────────────────────────────────────┬───────────────────────────────────┐
  │            報告内容の解釈            │            実際の状態             │
  ├──────────────────────────────────────┼───────────────────────────────────┤
  │ proxy.pac の配信を停止した           │ されていない（配信中）            │
  ├──────────────────────────────────────┼───────────────────────────────────┤
  │ プロキシの認証を無効化した           │ されていない（407                 │
  │                                      │ Basic認証が有効）                 │
  ├──────────────────────────────────────┼───────────────────────────────────┤
  │ ファイアウォールで直接通信を許可した │ 一部のみ（DIRECT_LIST該当分だけ） │
  ├──────────────────────────────────────┼───────────────────────────────────┤

  └──────────────────────────────────────┴───────────────────────────────────┘

  │ 137.4s      │ 60828 / 60831 │ A + AAAA │ 成功                │
  ├─────────────┼───────────────┼──────────┼─────────────────────┤
  │ 188.0s      │ 60828         │ A        │ 成功                │
  └─────────────┴───────────────┴──────────┴─────────────────────┘

  これらは全て WireGuard 
  クライアントがエンドポイント解決のために発したクエリです。

  根拠：
  - ソースポートが全て他の数十クエリと共有 — Windows DNS Client
  サービス（Dnscache）経由
  - tukushityann.net → 58.188.75.42 = WireGuard の接続先そのもの
  - 196秒目からWireGuard Handshake Initiation が 58.188.75.42:51820 に送出開始 —
   DNS解決の直後

  dig と Resolve-DnsName -Server のパケットが無い理由

  Resolve-DnsName -Server 指定時や dig は Windows DNS Client 
  サービスを経由せず、自前でソケットを開いてDNSクエリを直送します。
  そのパケットがキャプチャに無い = 物理NICから出ていないということです。

  原因として最も有力なのは：

  - WSL2 の dig: WSL2 は Hyper-V
  仮想マシンとして動作しており、物理イーサネットではなく 仮想NIC（vEthernet）
  経由で通信する。キャプチャ対象が物理NICのみの場合、WSL2からのパケットは映らな
  い。さらにWSL2のNAT/ファイアウォール層でドロップされた可能性もある
  - PowerShell の Resolve-DnsName -Server: -Server を指定すると DNS Client
  サービスのキャッシュ/転送を迂回して直接クエリを送信するが、Windows Defender
  ファイアウォール等で アプリケーション単位のアウトバウンドルール
  によりブロックされた可能性がある

  つまり

  Windows DNS Client (svchost.exe)  → 10.72.1.11:53  ✅ 通る → 応答あり
  Resolve-DnsName -Server           → 10.72.1.11:53  ❌ NICから出ていない
  WSL2 dig                          → 10.72.1.11:53  ❌
  仮想NIC経由で物理NICに到達していない

  Windows のバックグラウンドサービスは正常にDNS解決できているのに、手動コマンド
  だけ失敗するという矛盾は、パケットがそもそもワイヤに出ていなかったことで説明が
  つきます。当日は nslookup での確認（Windows DNS Client
  経由）も試してみると切り分けに役立つかもしれません。

> vpnを通すならsoftether vpnが安牌？                                            

● この環境では安牌だと思います。

  SoftEther が適している理由

  キャプチャから判明したこの環境の制約に対して：

  ┌────────────────────┬───────────┬───────────────────────────────────────┐
  │        制約        │ WireGuard │        SoftEther (HTTPS mode)         │
  ├────────────────────┼───────────┼───────────────────────────────────────┤
  │ UDP 外向きブロック │ ❌ 致命的 │ ⭕ TCP 443 を使うため無関係           │
  ├────────────────────┼───────────┼───────────────────────────────────────┤
  │ HTTP プロキシ強制  │ ❌ 非対応 │ ⭕ CONNECT メソッドでプロキシ越え可能 │
  ├────────────────────┼───────────┼───────────────────────────────────────┤
  │ Basic 認証         │ -         │ ⭕ プロキシ認証の設定欄がある         │
  └────────────────────┴───────────┴───────────────────────────────────────┘

  SoftEther の VPN over HTTPS は通常の HTTPS
  通信と見分けがつきにくく、プロキシの CONNECT
  を通してトンネルを張れるのが強みです。

  想定される構成パターン

  クレデンシャルが手に入る場合：
  Client → proxy01.osaka.hal.ac.jp:8080 (CONNECT + Basic認証) → SoftEther
  Server:443

  クレデンシャルが手に入らない場合：
  SoftEther サーバのドメインを DIRECT_LIST に該当させるか、直接 TCP 443
  が通るか次第。*.amazonaws.com がリストに入っているので、AWS
  上に建てれば直接通る可能性がある。

  注意点

  - TCP ベースなので WireGuard (UDP) と比べてレイテンシ・スループットは劣る
  - 会場のプロキシが DPI（Deep Packet Inspection）で CONNECT
  トンネルの長時間維持を切る設定だと途切れる可能性がある（が、教育機関のSquid系
  プロキシでそこまでやっている例は少ない）

> WireGuard tcp 443にできたりしない？                                           

● WireGuard 自体は UDP 専用で、プロトコルレベルで TCP に対応していません。

  ただし UDP を TCP 443 で包む方法はあります。

  現実的な選択肢

  ┌───────────┬───────────────────────────┬─────────────────────────────────┐
  │   方式    │   プロキシ CONNECT 越え   │          導入の手軽さ           │
  ├───────────┼───────────────────────────┼─────────────────────────────────┤
  │ wstunnel  │ ⭕ WebSocket              │ サーバ・クライアント両方に導入  │
  │           │ なのでプロキシ越え可能    │                                 │
  ├───────────┼───────────────────────────┼─────────────────────────────────┤
  │ udp2raw   │ ❌ 生 TCP                 │ Linux 限定                      │
  │           │ なのでプロキシ不可        │                                 │
  ├───────────┼───────────────────────────┼─────────────────────────────────┤
  │ bore /    │ ⭕ HTTP トンネル          │ WireGuard                       │
  │ chisel    │                           │ との組み合わせがやや煩雑        │
  └───────────┴───────────────────────────┴─────────────────────────────────┘

  wstunnel が一番素直

  Client                          Server (tukushityann.net)
  WireGuard                       WireGuard (:51820)
      ↓ UDP                           ↑ UDP
  wstunnel client                 wstunnel server (:443)
      ↓ WSS (TCP 443)                ↑
    [proxy CONNECT]  ──────────→  :443

  サーバ側：
  wstunnel server wss://0.0.0.0:443 --restrict-to 127.0.0.1:51820

  クライアント側：
  wstunnel client -L udp://127.0.0.1:51820:127.0.0.1:51820 \
    wss://tukushityann.net:443 \
    --http-proxy proxy01.osaka.hal.ac.jp:8080 \
    --http-proxy-login user:pass

  WireGuard の Endpoint を 127.0.0.1:51820 に向ければ、既存の WireGuard
  設定をほぼそのまま使えます。

  ただし

  - wstunnel を挟む分、WireGuard 単体より遅延が増える（TCP over TCP 問題に近い）
  - サーバ側で wstunnel を追加で動かす必要がある
  - 当日までにサーバ側の準備・テストが間に合うかどうか

  手軽さと安定性を取るなら SoftEther、既存の WireGuard 構成を活かしたいなら
  wstunnel、という判断になると思います。
